2. Configuring Siebel Applications

Overview of Siebel Tools and Workspaces

The Workspace model is the cornerstone of modern Siebel development (IP25+). It provides a safe, isolated, and auditable environment for developers to make repository changes without impacting the mainline application or interfering with each other. This enables a true parallel development methodology, managed entirely through the browser-based Siebel Web Tools.

  • MAIN Workspace: This is the top-level, read-only workspace. Think of it as the master, production-ready version of your repository. No development ever happens directly in MAIN.
  • Integration Workspace: This is a child workspace created from MAIN. It acts as a central branch for a major release or project. This is the level where changes from multiple developers are consolidated and tested together.
  • Developer Workspace: This is a child of an Integration Workspace. It's the personal sandbox where an individual developer performs all their configuration tasks. Changes made here are completely isolated until they are submitted and delivered.
    **Where to see it:** In Siebel Web Tools, the Workspace dashboard is the first thing you see, showing the hierarchy and allowing you to create or open your workspaces.

Basic configuration tasks

The workspace lifecycle enforces a structured and auditable process for promoting changes from a developer's sandbox to the mainline repository.

  • Open & Lock: A developer opens their workspace and then finds and locks the specific objects they need to modify (e.g., an Applet, a Business Component). Locking is crucial as it prevents another developer from editing the exact same object in a different workspace, thus preventing conflicts.
  • Checkpoint: Think of this as a "save point." Periodically, a developer 'Checkpoints' their workspace, which creates a new version of their changes. This allows them to revert to a previous state if a configuration goes wrong.
  • Submit for Delivery: When the developer has finished and tested their changes, they submit their workspace for review. This action signals that the work is complete and ready for integration.
  • Deliver: A release manager or administrator reviews the submitted workspace and, if approved, 'Delivers' it. Delivery is the process of merging the changes from the developer workspace into its parent Integration Workspace.

Creating and managing data models

The Siebel Data Model is the definition of how business information is structured and stored. It is a three-tiered logical model that is mapped to a physical database implementation, providing a crucial layer of abstraction.

  • Data Layer (Physical): This consists of Table and Column objects. These objects directly represent the physical schema of your database (e.g., the `S_OPTY` table and its columns like `OPTY_NAME`, `REVN_AMT`). You configure this layer in Siebel Tools.
  • Business Layer (Logical): This layer consists of Business Components (BCs). A BC provides a logical, business-centric representation of data. It can combine columns from multiple tables using joins and links to create a "virtual" record that makes sense to the business, like an "Opportunity."
  • UI Layer (Presentation): This layer consists of Applets and Views. These objects take the data exposed by the Business Components and present it to the end-user in a graphical format, such as a list or a form.

Data Model Architecture

Configuring business components and objects

Business Components (BCs) are the workhorses of the Siebel application. They define the data structure and business logic for entities. Business Objects (BOs) group related BCs to establish context for a view.

  • Fields: Represent individual data points (e.g., 'Name', 'Status'). Every field is mapped to a specific column in a database table.
  • Joins: A powerful feature used to bring in data from a related "lookup" table into a single BC. For example, a join on the Opportunity BC can pull in the Account Name from the Account table. This creates a flattened, denormalized view that is easy for the UI to display.
  • Links: These define the master-detail (parent-child) relationships between different Business Components within a Business Object. For example, a link defines how Siebel finds all the `Contacts` related to a specific `Account`.
  • Properties: BCs and their fields have hundreds of properties that declaratively control behavior, such as 'Read Only', 'Required', and 'Validation' expressions, minimizing the need for scripts.

Setting up application parameters

Application Parameters, known as System Preferences in the UI, are a simple and powerful way to control application behavior without making repository changes. They act as global constants that can be referenced throughout the application.

  • What they are: They are simple name-value pairs stored in the Siebel database. For example, you could have a parameter named `SessionTimeout` with a value of `30`.
  • Why they are used: They are perfect for storing values that might change between environments (like URLs for an external system) or for controlling features. A developer can write code that checks `TheApplication().GetProfileAttr("MyFeatureEnabled")` and if the value is `Y`, the feature is active.
  • How to manage them: They are managed in the live application via the `Administration - Application > System Preferences` view. This means you can change behavior on the fly without having to compile and deploy a new version of the repository.

System Preferences UI

Managing system preferences

Effectively managing system preferences is key to maintaining a flexible and configurable application. Best practices involve careful planning and documentation.

  • Adopt Naming Conventions: As the number of preferences grows, a clear naming convention is vital. Prefixing names can help group related preferences (e.g., `MyModule.Setting1`, `MyModule.Setting2`).
  • Mind the Cache: For performance, the Siebel server caches system preferences in memory. When you change a value in the UI, it doesn't take effect immediately for all users. An administrator must navigate to the `Administration - Application > System Preferences` view and select `Clear Cache` from the applet menu. This forces all servers to reload the latest values.
  • Migrate Your Changes: System Preferences are considered administrative data, not repository objects. This means they are not part of a workspace. To move them from a development environment to a test or production environment, you must include them in an Application Deployment Manager (ADM) package.

Creating and managing workflows

Siebel Workflow is a powerful automation engine that allows you to model and execute complex, multi-step business processes using a visual, drag-and-drop interface.

  • Process Designer: The graphical interface in Siebel Web Tools where you build your workflows. You drag different step types from a palette onto a canvas and connect them to define the process flow.
    **Where to see it:** In Web Tools, navigate to the `Workflow Process` object type in the Object Explorer.
  • Process Properties: These are the variables of your workflow. They hold data that can be passed from one step to another, allowing you to work with information as it moves through the process.
  • Key Step Types:
    • Siebel Operation: Allows you to query, insert, update, or delete records in a Business Component.
    • Decision Point: A diamond-shaped step that allows you to create branching logic (if/then/else) based on the value of a process property.
    • Business Service: A step that can call custom code or pre-built Siebel services to perform complex logic that goes beyond simple data operations.

Automating CRM processes

Siebel provides a suite of declarative (no-code) automation tools. Choosing the right tool for the job is a key architectural decision.

  • Workflow Processes: The most versatile tool. Best for orchestrating multi-step, long-running, or complex business processes that may involve branching, looping, and calls to external systems. Example: A multi-stage approval process for a large quote.
  • Assignment Manager: A highly specialized engine designed for one purpose: routing business data (like leads or service requests) to the right people or teams. It uses a sophisticated rules engine to evaluate criteria like territory, skills, or workload.
    **Where to see it:** Navigate to `Administration - Assignment` to view and define assignment rules.
  • State Models: The simplest automation tool, used to manage the lifecycle of an object. It defines the valid states (e.g., 'Open', 'Pending', 'Closed' for a Service Request) and enforces which transitions between states are allowed.
    **Where to see it:** Navigate to `Administration - Application > State Models`.

Customizing the user interface

The Siebel UI is configured by modifying Applet and View objects. The Web Layout Editor in Siebel Web Tools is the primary interface for visually designing the user experience.

  • Web Templates: These are the foundational HTML files that define the structure and layout of applets and views. They contain special Siebel tags (``, ``) that act as placeholders where the Siebel Web Engine will render controls and applets.
  • Web Layout Editor: This is a visual "drag-and-drop" tool in Web Tools. It shows the placeholders from the web template and allows you to map repository objects to them. You can drag controls (representing BC fields) onto the applet grid or drag entire applets onto the view layout.
  • Applet-to-BC Mapping: Each applet is tied to a single Business Component. The controls you place on the applet are then mapped to specific fields on that BC. This three-way link (Template Placeholder -> Applet Control -> BC Field) is what makes the data appear on the screen.

Personalizing user experiences

Siebel allows for a high degree of personalization to tailor the user experience to individual needs without requiring code changes.

  • User Preferences: End-users have control over their own experience. They can set personal preferences like their default startup view, time zone, and, most commonly, choose which columns to display in list applets and in what order.
    **How to validate:** In any list applet, right-click the column header and choose "Columns Displayed" to change your personal layout.
  • Personalization Rules: Administrators can create more powerful, rule-based personalization. These rules can dynamically show, hide, or format applets and controls based on user attributes like their role, location, or language. This ensures the UI is always relevant to the user's context.
  • Saved Queries: A simple but powerful feature. Users can save frequently used search criteria as a named query. This allows them to quickly access specific subsets of data with a single click from the "Saved Queries" dropdown in any applet.
