3. Managing Customer Data

Ensuring data accuracy and completeness

Maintaining data integrity is crucial. Siebel ensures data quality at the point of entry using powerful declarative rules defined on the Business Component Field object.

Key Validation Mechanisms:

  • Required Flag: Setting the **Required** property on a Field to `TRUE` ensures the user provides a value before the record is saved. The UI shows an asterisk (*) next to the field name.
  • Validation Property: This property contains a logical expression that must evaluate to true when the record is saved. If false, the system prevents the save and displays a customizable error message.
  • Validation Scope: Validation rules can only reference **other fields within the same Business Component record** (e.g., checking if one date is after another date in the same record).
  • Picklists: Using a **bounded** picklist restricts user input to a predefined set of values, ensuring consistency on fields like 'Status' or 'Type'.

Field Validation Rule Example

Example: A validation rule might be defined on an Opportunity's **Probability** field: `[Probability] >= 0 AND [Probability] <= 100`. If violated, the user receives an error message and the save is aborted.

Data cleansing techniques

While validation prevents bad data from entering, **Data Cleansing** addresses data that is already in the system, primarily focusing on **de-duplication**.

De-Duplication Process:

  • Matching Criteria: Administrators configure rules in Siebel to define what constitutes a duplicate (e.g., two contacts with the same name and phone number).
  • Batch Mode: The cleansing process often runs in **batch mode** (via a Siebel server component) to scan the entire database for potential duplicates against the defined criteria.
  • Review and Merge: When duplicates are found, they are flagged. Users review the records and utilize the **Merge** functionality to combine them into one single **master record**, preserving related historical data (Activities, SRs) from all merged records.

Data Cleansing Process Flow

Validation: In the Siebel Web Client, search for a duplicate account or contact. Select the suspect record, click the Menu button in the applet, and select Check for Duplicates to run a real-time check.

Importing and exporting data

Enterprise Integration Manager (EIM) is the mandatory server component for bulk moving user data into or out of the Siebel application. EIM moves large volumes of data in batch mode.

The EIM Data Flow:

  1. Data Staging: External data is loaded into Siebel Interface Tables (e.g., `EIM_ACCOUNT`) using SQL. These tables act as a buffer.
  2. EIM Processing: The EIM component processes the staged data, resolving Foreign Keys (FKs) using User Keys, generating ROW\_IDs, and loading the data into Siebel Base Tables.
  3. Validation: EIM updates the `IF_ROW_STAT` column in the interface tables with the final status (`IMPORTED`, `FOREIGN_KEY`, etc.).
Crucial Rule: You must not use direct SQL to insert or update user data in Siebel Base Tables! This bypasses the generation of ROW\_IDs and the maintenance of Foreign Keys, corrupting data integrity.

EIM Architecture

Validation: In Siebel Tools, check the `EIM Interface Table` report to see how an interface table (e.g., `EIM_ACCOUNT`) maps to one or more core Siebel Base Tables (e.g., `S_ORG_EXT`).

Defining and managing customer segments

Customer Segmentation is the practice of dividing customers into groups based on characteristics like purchase history or demographics, typically within the Marketing module.

Segmentation in Practice:

  • Criteria-Based Lists: Segments are defined using complex search criteria across various BCs (e.g., find all Contacts in the "Finance" industry who bought a "Premium" product).
  • Dynamic Segments: The list of customers belonging to a segment is not fixed; it is **re-evaluated** whenever the segment is used. This ensures campaigns always target the most current audience.
  • Access Control for Master Data: This process uses Catalogs and Categories to organize master data and Access Groups (collections of Positions/Orgs/User Lists) to limit which employees can see specific products.

Master Data Access (Catalogs/Categories)

Validation: In the Siebel Web Client, navigate to Administration - Catalog > Catalogs. A Catalog or Category set to **Private** must have explicit access granted via a child Access Group.

Using segmentation for targeted marketing

Segmentation is the bridge between market research and campaign execution, enabling **targeted marketing** and reliable ROI measurement.

Targeted Campaign Flow:

  • Audience Selection: Marketers associate a defined **Segment** (or multiple segments) with a **Campaign**.
  • Channel Execution: The campaign is executed across chosen channels (email, SMS, call center list) targeting only the members of the segment.
  • Closed-Loop Analysis: Customer responses are tracked back to the specific campaign and segment that generated them. This closes the loop and allows for accurate calculation of campaign effectiveness.

Targeted Marketing Flow

Validation: Navigate to the Campaigns screen. Drill down on an active campaign and check the **Segments** view to see which target audiences are associated with it.

Contact and account management

Accounts and Contacts form the core of the customer master data. They are specialized **Party Business Components**, which means their data is structured differently in the database to allow for complex and dynamic relationships.

The Party Data Model:

  • S\_PARTY Base Table: Siebel uses the small, universal **S\_PARTY** table as a central linking mechanism for all party types (Contacts, Accounts, Positions, Organizations).
  • Extension Tables: The actual customer details are stored in specific **Extension Tables** (e.g., `S_ORG_EXT` for Accounts), which are linked back to `S_PARTY` using an **Implicit Join** on the `PAR_ROW_ID` column.
  • Flexibility: This structure makes it easy to model complex relationships (e.g., one Contact working for multiple Accounts).

Party Business Component Structure

Validation: In Siebel Tools, query **Business Component :: Contact**. Note that core fields map to the `S_CONTACT` extension table, linked via an **Implicit Join** on the `PAR_ROW_ID` field.

Creating and managing service requests

Service Requests (SRs) are tracked through a controlled State Model to ensure predictable and consistent handling of customer issues.

The State Model:

  • States and Transitions: The model defines a set of allowed States (e.g., Open, Pending, Closed) and the permissible Transitions between them.
  • Restrictions: Transitions can be restricted based on who performs the action (e.g., only a Call Center Manager can transition → Closed) or conditions (e.g., transition → Pending only if a field technician is assigned).
  • Administration: State Models are configured and deployed in the **Siebel Web Client** (Administration - Application > State Models), meaning changes do **not** require a full application deployment.

Service Request State Transitions

Validation: In the Siebel Web Client, navigate to Administration - Application > State Models. Select the Service Request object. In the Transitions view, you can check the Public flag and the Authorized Positions list to see who is allowed to perform each state change.

Handling customer cases and issues

Siebel provides specialized tools to assist service agents in resolving cases efficiently, going beyond simple SR logging.

Resolution Tools:

  • Solutions/Knowledge Base: A repository of common problems and their documented resolutions. Agents search this knowledge base and attach relevant solutions to the SR.
  • Entitlements: Used to determine the level of service a customer is due based on their Service Contract. This check ensures the customer receives the correct priority and resource allocation.
  • Activities: All actions taken by the agent (calls, emails, follow-up tasks) are logged as **Activities** associated with the SR record, ensuring an auditable history.

Service Request with Solutions Tab

Validation: Navigate to the Service screen, drill down on an SR, and select the Solutions view tab. You can search the knowledge base for a relevant article to attach to the SR.
