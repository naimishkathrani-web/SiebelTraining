What is Customer Relationship Management?

Customer Relationship Management (CRM) is the practice and system used to capture, manage, and analyze all interactions with customers across sales, service, and marketing.

Concept — in plain words

CRM stores customer contact details, tracks interactions (calls, emails, orders), manages support cases, and helps sales teams manage opportunities. It centralizes information so any employee can quickly understand a customer's history.

Example (telecom)

A customer calls with a billing issue. The service agent opens the customer record, sees the subscription details, prior complaints, and recent invoices — all in one place. This saves time and improves satisfaction.

Why it matters (reasoning)

Without a central system, information is fragmented across spreadsheets and emails. A CRM ensures consistent, auditable customer interactions and faster cross-team collaboration.

Business impact

  • Shorter case resolution times
  • Better conversion of leads to sales
  • Targeted campaigns increase marketing ROI

Importance of CRM in Business

CRM is a strategic asset — it helps businesses scale customer engagement while maintaining consistent service and measurable outcomes.

Detailed explanation

Companies use CRM to standardize customer-facing processes, measure performance, and provide a single source of truth. This matters especially when many teams touch a customer — sales, support, marketing, billing — and decisions must be fast and informed.

Example: Campaign conversion

Marketing runs a promotional campaign targeted by customer segment. With CRM segments and history, the team identifies likely responders and measures conversion, allowing continuous improvement of campaigns.

Key takeaway: CRM transforms customer data into actionable business decisions — increasing revenue and reducing churn.

Siebel CRM — High-level Overview

This application is designed to manage enterprise customer lifecycles with configurable modules for industry solutions such as eCommunications.

How the product is organized

The system exposes modules (Sales, Service, Marketing) composed of Views, Applets, Business Components and Business Objects. Administrators configure the UI and logic through the browser-based Web Tools and manage runtime behavior via system preferences and parameters.

Example (vertical)

In the eCommunications vertical, there are specialized screens for subscriber management and service orders. These screens reuse core components but add domain-specific fields and flows.

Estimated time: 12 minutes

Architecture — High Level

At a glance, the system separates data storage (database), application logic (server), and presentation (web client). Integration components connect external systems.

Components

  • Database (stores application data)
  • Application servers (object managers, workflow engine)
  • Web server & web client (user interface)
  • Integration layer (EAI / REST / SOAP adapters)

Why this design?

Separation improves scalability: you can scale web servers for many users and scale the database independently for data throughput.

Understanding the Data Model — Simple View

Focus on Business Components (BCs) and how they map to database tables. Think of BCs as the interface between UI and data.

Concept

Each BC exposes fields for an applet. A BC may use one or more tables. Joins and search specs determine which rows appear in a list or form.

Example

The Contact BC maps to the contact table and presents name, phone, email fields to the UI. If you add a custom column to the BC and map it to an extension table, applets can display it without database redesign.

Screen, View, Applet — UI Building Blocks

Screens group Views; Views contain Applets. Applets are the visible UI elements showing lists or forms.

Detailed explanation

Applets can be list, form, or multi-row. Views determine which applets and toolbars appear together. Screens are the runtime containers presented to responsibilities.

Example

An Account Screen might include a list applet showing related contacts and a form applet showing account details; together they allow users to view and act on account information efficiently.

Setting up Users and Roles

Users represent people; Responsibilities determine UI access; Access Groups manage server routing and data access.

Admin screens

Administrators create User records and assign a Responsibility. A Responsibility includes a list of Views and a default Home layout. Access Groups determine which server processes a user can use.

Example (scenario)

When onboarding a new service agent, create a User, assign a Service Responsibility and ensure the Home page shows the case queue and SLAs relevant to their team.

Responsibilities & Access Control

Responsibilities are the primary way access to screens and features is granted. Combine them with data-level security for tighter control.

How to think about it

Think of Responsibilities as job roles that determine which menus, views and actions a user can see. Access Groups control session behavior and Object Manager selection.

Business relevance

Proper responsibility setup avoids exposing sensitive data and reduces user confusion by presenting only necessary screens.

The Typical User Workflow

A typical sequence: Login → Open Home → Check queues → Open record → Work case/opportunity → Update and close.

Detailed steps

  1. Login and review task list
  2. Pick highest priority item in queue
  3. Follow the guided process for resolution or qualification
  4. Log the interaction and update the case/opportunity

Reasoning

Standardizing workflows ensures predictable outcomes and easier measurement of team performance.

Workspaces — Developer Model

Workspaces let developers and administrators make changes in isolation and then deliver them into the shared environment after testing.

How it works

Create a Dev Workspace, make UI or object changes there, test, and then deliver to Integration and MAIN. This reduces change collisions and supports parallel work.

Example

An administrator creates a new Applet in her Dev Workspace and tests it with sample data. After approval, she delivers to the Integration workspace for QA, then to MAIN for availability to users.
