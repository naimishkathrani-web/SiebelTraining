Day 1 Begins

What is customer relationship management?

Customer Relationship Management (CRM) is a business strategy and technology platform focused on managing all interactions with customers and prospects. The core purpose is to maximize the value of every customer relationship.

CRM Objectives:

  • Acquisition: Winning new customers by effective lead management and nurturing sales opportunities. This initial phase focuses on converting prospects into clients.
  • Retention: Keeping existing customers loyal through excellent service and personalized communication. Long-term customer satisfaction is key to business stability.
  • Profitability: Increasing the lifetime value of each customer by understanding their needs and selling them relevant products and services. This ensures that relationships are valuable to the company.

Siebel enforces this with a single database to house all customer information, guaranteeing data consistency across all user interactions and communication channels (Web, Call Center, Mobile).

CRM Lifecycle Diagram

Analogy: The CRM lifecycle is a continuous feedback loop: you acquire a customer, serve them, analyze their needs, and then market new offerings, which leads to repeat business and retention.

Importance of CRM in business

A CRM system acts as the central nervous system for a customer-centric business. It breaks down information silos to ensure efficient, collaborative customer engagement.

Business Value:

  • Unified View (360-degree): Gives every employee-from sales to service-a complete, real-time history of the customer, preventing customers from having to repeat themselves.
  • Data Consistency: All information resides in a single source, eliminating data silos and ensuring reliable reporting and planning.
  • Process Automation: Automating repetitive administrative tasks (like lead assignment or service ticket routing) frees employees to focus on strategic, high-value tasks, like selling and problem-solving.

CRM Benefits Diagram

Benefit: Siebel helps managers make better decisions with a real-time view of the entire sales team and organization.

Key features and benefits

Siebel's platform is designed for enterprise scale, focusing on maximum customization through metadata, not code.

Core Technical Advantages:

  • Declarative Configuration (Metadata): The entire application structure (UI, logic, fields) is defined by metadata in the repository. This means complex logic is implemented by setting properties in Siebel Web Tools, not by writing source code.
  • Cloud Native Architecture: Modern Siebel architecture (IP25 concepts) uses containerized, immutable services. This enables all changes to be versioned, deployed automatically, and supports zero-downtime upgrades.
  • Open UI & Mobile: The interface is built on standard web technologies (HTML5) and supports responsive design, making it intuitive for mobile use and easily customized.

Siebel Features Diagram

Insight: The architecture prioritizes Automation (fully automated deployment) and Service Orientation (independent, microservices) for agility and modern operations.

Navigating Siebel CRM

The Siebel Web Client UI is structured hierarchically, defined by the UI Layer of the architecture. Understanding this structure is essential for finding and validating data.

Navigation Hierarchy:

  • Screens: The highest level, representing major functional areas (e.g., Accounts, Opportunities). Accessed via the main tabs.
  • Views: Groups of related data within a Screen (e.g., Account List View, Account Detail View). Views are accessed via the Link Bar or View Tabs below the main screen tabs.
  • Applets: The actual containers of data within a View. They display records in two forms: a List Applet (multiple records in a grid) or a Form Applet (a single record in a detailed layout).
  • Drilldown: A hyperlink on a record that quickly takes you to a more detailed view of that specific record.

Siebel UI Navigation Map

Validation: In the Application, navigate to the Accounts tab (Screen), click the Accounts List link (View), and observe the grid of data (List Applet). Click on any Account Name (Drilldown) to switch to the Account Form Applet.

Setting up users and roles

Siebel's security model separates functional access (what you can do) from data access (what records you can see) using a multi-layered hierarchy.

Functional Access (The Key):

  • Responsibilities: A Responsibility is a collection of allowed Views. It determines the functional permissions of a user. The Developer Responsibility is required for configuration access.

Data Access (The Badge):

  • Positions: Every Employee holds one or more Positions. Position determines data ownership.
  • Organizations: Positions belong to an Organization (a Business Unit). Organization-based security limits data visibility across large segments.
  • Position Hierarchy: Defines the reporting lines, enabling managers to see data owned by their direct and indirect reports.

Functional vs. Data Access Control

Analogy: A Responsibility is your key to the building (access to views), but your Position/Organization is your badge that limits which office doors (records) you can open once inside. These two mechanisms are independent.

Overview of Siebel tools

Configuration uses declarative techniques (setting properties) to implement business logic. Siebel Web Tools is the modern IDE used to modify **object definitions** which reside in the Siebel repository.

Siebel Tools and the Repository:

  • Declarative Configuration: The core principle: implementing logic by modifying metadata object properties, not by writing custom code or modifying source files.
  • Object Definitions: These are the metadata records (e.g., Business Component, Field, Applet) that define the Siebel application.
  • Repository: This is the database table where the metadata (object definitions) lives. Modern Siebel versions rely on consuming data directly from the Repository table.

Tools / Runtime Architecture

Runtime: For modern deployments, Application Object Managers consume object definitions directly from the database repository, making deployment faster and removing the dependency on manually compiled SRF files.

Basic configuration tasks

The Workspace Model is essential for parallel development, enabling multiple developers to work concurrently on the repository without overwriting each other's changes.

Workspace Development (Concurrent Model):

  • Sandbox (Developer Workspace): Every developer works in their own personal workspace (e.g., `DEV_TRAINEE##_`). This is a sandbox where configurations are isolated from the main application.
  • Locking: When a developer modifies an object, that specific object instance is locked within their workspace, ensuring exclusive write access and preventing merge conflicts.
  • Submit for Delivery: When development is complete, the workspace status is set to **Submitted for Delivery**.
  • Deliver: An administrator reviews and **Delivers** the changes, merging them into a parent **Integration Workspace** and creating a new object version.

Workspace Hierarchy and Flow

Isolation: Workspaces are hierarchical: **MAIN** $\to$ **Integration** $\to$ **Developer**. Each developer only sees the incremental changes (the **delta**) within their branch.

Creating and managing data models

The configuration process adheres to the Bottom-Up strategy for execution: start with the Data Layer, move to the Business Layer, and finish with the UI Layer.

The Data Layer (Tables/Columns):

  • Data Model: Siebel uses a **Party Data Model** (`S_PARTY`) and a structured set of base tables (`S_OPTY`, `S_SRV_REQ`).
  • Custom Fields: Best practice is to reuse existing 1:1 Extension Tables (like `S_OPTY_X`) for custom fields. Custom column names must be prefixed with **X\_**.
  • ROW\_ID: This is the fundamental, universally unique **Primary Key (PK)** for every record, maintained automatically by Siebel.

The Business Layer (BCs/Fields):

  • Field Mapping: A Field object in the BC (e.g., `Customer Score`) maps directly to a physical Column in the data layer.
  • Calculated Fields: Fields derived from an expression (e.g., `[Revenue] * 0.1`). These **do not map to a column** and are always read-only.

Data Model Architecture

Validation: In Siebel Web Tools, use the **Object Explorer (OE)** to view Table $\to$ Column to see the physical layer, and Business Component $\to$ Field to see the logical layer.

Configuring business components and objects

Configuration uses declarative logic applied directly to BC properties, controlling data behavior.

Data Control and Validation:

  • Search Specification: A permanent `WHERE` clause on the BC that filters the data returned to the application (e.g., only showing records where `STATUS = 'Active'`).
  • Validation: A field property that defines an expression that must be `TRUE` when a record is saved. This prevents bad data and enforces simple business rules.
  • Joins: An object in the BC that brings in fields from a secondary table (the "joined table") based on a foreign key relationship, creating a single virtual record.
  • User Properties: Named arguments added to a BC or Field that implement complex IF/THEN logic (e.g., dynamically making a field read-only based on a status flag) without using code.

BC Field Join Mapping

Relationship: The Field object's Join property links it to the Join object, which in turn links to the physical joined table. The field's Column property specifies which column to fetch from that joined table.

Setting up application parameters

Application Parameters (System Preferences) are global variables stored directly in the Siebel database. They provide a vital separation between global variables and the static repository metadata.

Key Advantages of Parameters:

  • Dynamic Control: Values can be updated in the Siebel Web Client (Administration - Application > System Preferences) without requiring any compiler or deployment cycle. This is used for quick changes (e.g., switching an external server URL).
  • Profile Attributes: These parameters are often referenced in business logic (Workflows, Scripts) using the function `GetProfileAttr("ParameterName")` to dynamically control application behavior.
  • Environment Separation: They hold values that differ between environments (Dev vs. Prod), making it easier to promote the core application code without manual changes.

Managing system preferences

System Preferences are cached in memory by the Application Object Manager on the Siebel Server to ensure fast access. This requires a crucial step after any modification.

The Cache Management Rule:

  • Caching: Siebel clients and server components cache the preference values upon login or startup. A change in the database isn't instantly visible to running user sessions.
  • Clearing the Cache: After updating a parameter value in the **System Preferences** view, an administrator must select Menu > Clear Cache from the applet. This action forces all active server components to reload the latest values from the database.

System Preferences Clear Cache

Validation: After clicking Clear Cache, a running Workflow that uses the parameter should immediately execute with the new value, confirming the server-side memory refresh.

Creating and managing workflows

A Workflow Process is a declarative (no-code) way to automate complex, multi-step business logic and integrate different application features.

Core Workflow Concepts:

  • Process Properties: Variables that store data throughout the workflow's execution (e.g., the `Object Id` of the record that triggered the flow).
  • Workflow Steps: The building blocks of the process, including: Start, End, Siebel Operation, Business Service, and Decision Point steps.
  • Business Object (BO) Context: Workflows that perform operations on business components must be assigned a Business Object (e.g., the `Contact` BO) to provide context for the Business Components they manipulate.

Workflow Designer Interface

Building: Steps are dragged from the Palette onto the Workspace and connected using Connectors to define the process sequence.

Workflow Step: Siebel Operation

The Siebel Operation step is essential for direct manipulation of Siebel data (CRUD actions) within a workflow. It operates on a specified Business Component.

Configuration Focus:

  • Operation Types: Supports Insert, Update, Delete, or Query.
  • Query Output: For a Query, the Output Arguments tab is used to map specific BC Fields (e.g., `Account Name`) to Process Properties so the data is accessible in later steps.
  • Update Input: For an Update, the Input Arguments tab defines which fields should be set to which values (literal or process property).

Siebel Operation Step Properties

Logic: The step uses the Search Specification in the Input Arguments (if querying) to define the `WHERE` clause for the database retrieval.

Workflow Step: Decision Point

The Decision Point step, represented by a diamond shape, is used to introduce branching logic (IF/THEN/ELSE) into the workflow path.

Configuration Details:

  • Conditional Connectors: Each path leading out of the step (except the default) is a connector that must have a specific condition defined based on a Process Property (the input).
  • Compose Condition Criteria: This dialog box is used to visually build the logical expression (e.g., `[Middle Name] IS NOT NULL`).
  • Default Branch: One connector must remain without any condition defined. This is the default path the workflow will take if all other defined conditions evaluate to false.

Decision Point Step Properties

Execution: The workflow engine evaluates conditions in a defined order and follows the path of the first condition that evaluates to true. If none are true, it takes the default path.

Customizing the user interface

UI customization is the process of modifying the look and feel of the Siebel application by manipulating the relationship between Web Templates and Applets/Views.

Web Templates:

  • HTML Files: Web Templates are essentially **HTML files** (`.swt`) that define the layout and formatting of Applets and Views. They contain standard HTML tags mixed with special **Siebel Tags** (``) that act as placeholders for dynamic data.
  • Web Layout Editor: The visual tool used to **map** Applet controls and list columns to the placeholder IDs within the Web Template.
  • Binding: This mapping process creates an `Applet Web Template Item` object in the repository, linking the logical object (Applet, Control) to the physical layout location.

Web Layout Editor Interface

Action: The editor allows you to drag a Control (which links to a BC Field) from the palette onto a Placeholder on the grid, defining where that field will appear in the running application.

Personalizing user experiences

Siebel allows users to tailor the interface to their specific needs, often without requiring administrative intervention.

User-Driven Customization:

  • User Preferences: Users can set individual preferences like their default **startup view**, **time zone**, and **default spell check options** from the application-level **Tools** menu.
  • Customizing List Applets: End-users can permanently change the order of columns, show/hide columns, and set **column widths** in list applets to optimize their working view.
  • Saved Queries: Users can save frequently used search filters for easy recall from the **Saved Queries** drop-down list on any Applet.

Personalization Example

Validation: In the Siebel Web Client, go to Tools > User Preferences. The screen that appears allows you to set preferences like the Default View and Time Zone.

Assignment 5: Building a Simple Workflow

Task: Create a Contact Status Update Workflow

  1. In Siebel Tools, create a new **Workflow Process** named `TRAINEE##_Update Contact Status` in your project. Set its **Business Object** to `Contact`.
  2. Add a **Start** step and an **End** step.
  3. Insert a **Siebel Operation** step named `Query Contact` after Start. Configure it for the `Contact` BC with the **Operation** `Query`.
  4. Set the **Input Arguments** for `Query Contact`: Map the **Search Spec** to the Process Property **Object Id**. This retrieves the record the workflow is run against.
  5. Insert another **Siebel Operation** step named `Update Status` after `Query Contact`. Configure it for the `Contact` BC with the **Operation** `Update`.
  6. Set the **Input Arguments** for `Update Status`: Set the field `Status` to the **Literal Value** `"Active"`.
  7. Connect the steps, **Validate** the workflow, then **Publish** and **Activate** it via the Siebel Web Client.

Day 1 Review and Q&A

Key Takeaways for Configuration:

  • Architecture: Everything flows through the **Business Component** layer, which abstracts the **Data Layer** from the **UI Layer**.
  • Change Management (IP25): Development occurs in an isolated **Developer Workspace**. Changes are delivered by **Submitting** and **Delivering** to an Integration Workspace.
  • Data Layer: The **ROW\_ID** is the universal Primary Key (PK). Custom columns must use the `X_` prefix.
  • Automation: Workflows use **Steps** and **Process Properties**, and are made live via **Publish and Activate** (as they are not compiled into the SRF).
