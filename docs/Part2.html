Day 2 Begins

Ensuring data accuracy and completeness

Maintaining data integrity is crucial. Siebel ensures data quality at the point of entry using powerful declarative rules defined on the Business Component Field object.

Key Validation Mechanisms:

  • Required Flag: Setting the **Required** property on a Field to `TRUE` ensures the user provides a value before the record is saved. The UI shows an asterisk (*) next to the field name.
  • Validation Property: This property contains a logical expression that must evaluate to true when the record is saved. If false, the system prevents the save and displays a customizable error message.
  • Validation Scope: Validation rules can only reference **other fields within the same Business Component record** (e.g., checking if one date is after another date in the same record).
  • Picklists: Using a **bounded** picklist restricts user input to a predefined set of values, ensuring consistency on fields like 'Status' or 'Type'.

Field Validation Rule Example

Example: A validation rule might be defined on an Opportunity's **Probability** field: `[Probability] >= 0 AND [Probability] <= 100`. If violated, the user receives an error message and the save is aborted.

Data cleansing techniques

While validation prevents bad data from entering, **Data Cleansing** addresses data that is already in the system, primarily focusing on **de-duplication**.

De-Duplication Process:

  • Matching Criteria: Administrators configure rules in Siebel to define what constitutes a duplicate (e.g., two contacts with the same name and phone number).
  • Batch Mode: The cleansing process often runs in **batch mode** (via a Siebel server component) to scan the entire database for potential duplicates against the defined criteria.
  • Review and Merge: When duplicates are found, they are flagged. Users review the records and utilize the **Merge** functionality to combine them into one single **master record**, preserving related historical data (Activities, SRs) from all merged records.

Data Cleansing Process Flow

Validation: In the Siebel Web Client, search for a duplicate account or contact. Select the suspect record, click the Menu button in the applet, and select Check for Duplicates to run a real-time check.

Importing and exporting data

Enterprise Integration Manager (EIM) is the mandatory server component for bulk moving user data into or out of the Siebel application. EIM moves large volumes of data in batch mode.

The EIM Data Flow:

  1. Data Staging: External data is loaded into Siebel Interface Tables (e.g., `EIM_ACCOUNT`) using SQL. These tables act as a buffer.
  2. EIM Processing: The EIM component processes the staged data, resolving Foreign Keys (FKs) using User Keys, generating ROW\_IDs, and loading the data into Siebel Base Tables.
  3. Validation: EIM updates the `IF_ROW_STAT` column in the interface tables with the final status (`IMPORTED`, `FOREIGN_KEY`, etc.).
Crucial Rule: You must not use direct SQL to insert or update user data in Siebel Base Tables! This bypasses the generation of ROW\_IDs and the maintenance of Foreign Keys, corrupting data integrity.

EIM Architecture

Validation: In Siebel Tools, check the `EIM Interface Table` report to see how an interface table (e.g., `EIM_ACCOUNT`) maps to one or more core Siebel Base Tables (e.g., `S_ORG_EXT`).

Defining and managing customer segments

Customer Segmentation is the practice of dividing customers into groups based on characteristics like purchase history or demographics, typically within the Marketing module.

Segmentation in Practice:

  • Criteria-Based Lists: Segments are defined using complex search criteria across various BCs (e.g., find all Contacts in the "Finance" industry who bought a "Premium" product).
  • Dynamic Segments: The list of customers belonging to a segment is not fixed; it is **re-evaluated** whenever the segment is used. This ensures campaigns always target the most current audience.
  • Access Control for Master Data: This process uses Catalogs and Categories to organize master data and Access Groups (collections of Positions/Orgs/User Lists) to limit which employees can see specific products.

Master Data Access (Catalogs/Categories)

Validation: In the Siebel Web Client, navigate to Administration - Catalog > Catalogs. A Catalog or Category set to **Private** must have explicit access granted via a child Access Group.

Using segmentation for targeted marketing

Segmentation is the bridge between market research and campaign execution, enabling **targeted marketing** and reliable ROI measurement.

Targeted Campaign Flow:

  • Audience Selection: Marketers associate a defined **Segment** (or multiple segments) with a **Campaign**.
  • Channel Execution: The campaign is executed across chosen channels (email, SMS, call center list) targeting only the members of the segment.
  • Closed-Loop Analysis: Customer responses are tracked back to the specific campaign and segment that generated them. This closes the loop and allows for accurate calculation of campaign effectiveness.

Targeted Marketing Flow

Validation: Navigate to the Campaigns screen. Drill down on an active campaign and check the **Segments** view to see which target audiences are associated with it.

Contact and account management

Accounts and Contacts form the core of the customer master data. They are specialized **Party Business Components**, which means their data is structured differently in the database to allow for complex and dynamic relationships.

The Party Data Model:

  • S\_PARTY Base Table: Siebel uses the small, universal **S\_PARTY** table as a central linking mechanism for all party types (Contacts, Accounts, Positions, Organizations).
  • Extension Tables: The actual customer details are stored in specific **Extension Tables** (e.g., `S_ORG_EXT` for Accounts), which are linked back to `S_PARTY` using an **Implicit Join** on the `PAR_ROW_ID` column.
  • Flexibility: This structure makes it easy to model complex relationships (e.g., one Contact working for multiple Accounts).

Party Business Component Structure

Validation: In Siebel Tools, query **Business Component :: Contact**. Note that core fields map to the `S_CONTACT` extension table, linked via an **Implicit Join** on the `PAR_ROW_ID` field.

Creating and managing service requests

Service Requests (SRs) are tracked through a controlled State Model to ensure predictable and consistent handling of customer issues.

The State Model:

  • States and Transitions: The model defines a set of allowed States (e.g., Open, Pending, Closed) and the permissible Transitions between them.
  • Restrictions: Transitions can be restricted based on who performs the action (e.g., only a Call Center Manager can transition $\to$ Closed) or conditions (e.g., transition $\to$ Pending only if a field technician is assigned).
  • Administration: State Models are configured and deployed in the **Siebel Web Client** (Administration - Application > State Models), meaning changes do **not** require a full application deployment.

Service Request State Transitions

Validation: In the Siebel Web Client, navigate to Administration - Application > State Models. Select the Service Request object. In the Transitions view, you can check the Public flag and the Authorized Positions list to see who is allowed to perform each state change.

Handling customer cases and issues

Siebel provides specialized tools to assist service agents in resolving cases efficiently, going beyond simple SR logging.

Resolution Tools:

  • Solutions/Knowledge Base: A repository of common problems and their documented resolutions. Agents search this knowledge base and attach relevant solutions to the SR.
  • Entitlements: Used to determine the level of service a customer is due based on their Service Contract. This check ensures the customer receives the correct priority and resource allocation.
  • Activities: All actions taken by the agent (calls, emails, follow-up tasks) are logged as **Activities** associated with the SR record, ensuring an auditable history.

Service Request with Solutions Tab

Validation: Navigate to the Service screen, drill down on an SR, and select the Solutions view tab. You can search the knowledge base for a relevant article to attach to the SR.

Managing sales pipelines and opportunities

The core of Siebel Sales is **Opportunity Management**, which uses a structured sales pipeline to guide potential deals from lead to closed/won or lost status.

Opportunity Attributes:

  • Sales Stage: Defines the current phase of the deal (e.g., 01-Prospecting, 03-Closing, 05-Won). This often drives the expected timeline and resources required.
  • Probability: A key metric that is often driven by the Sales Stage. This is used for forecasting.
  • Revenue: Tracks the expected or committed value of the deal.
  • Sales Team (MVG): An Multi-Value Group (MVG) that links the Opportunity to multiple **Positions** (people) involved in the sale.

Opportunity Sales Pipeline View

Validation: Navigate to the Opportunities screen and view the Sales Stage column. Notice how the values are typically numbered to enforce a chronological sales flow.

Forecasting and quota management

The Siebel Forecasting module allows management to collect, consolidate, and analyze projected sales data against assigned targets.

Key Concepts:

  • Hierarchical Roll-up: Forecasts submitted by individual Sales Reps are automatically **aggregated** up the **Position Hierarchy** to provide sales managers and executives a consolidated view of projected revenue for their entire reporting structure.
  • Forecasting Periods: The system allows for setting targets (quotas) and collecting forecasts based on predefined time periods (e.g., quarterly or annual).
  • Data Model: Forecasting data uses dedicated tables and BCs (e.g., `S_FORECAST`), allowing for separation from the core Opportunity records.

Sales Forecasting Flow

Validation: Navigate to the Forecast screen. You can see how individual forecasts at the lowest level are consolidated into a larger view (e.g., a Regional Manager view) based on the reporting structure.

Creating and managing marketing campaigns

Siebel's campaign management is geared toward multi-stage, multi-channel execution and detailed response tracking for accurate ROI.

Campaign Lifecycle:

  • Design: Campaigns are designed to target specific **Segments** (audiences) and execute a series of steps (e.g., sending an initial email, followed by a follow-up call if no response).
  • Execution: Campaigns interact with various channels (Web, email, print) to deliver personalized offers.
  • Response Tracking: The system tracks customer response (e.g., a purchase, a form fill) and links it directly back to the originating Campaign ID and the Segment ID.
  • Lead Creation: A successful campaign response can automatically create a **Lead** record, passing all relevant data to the sales process.

Marketing Campaign Flow

Validation: Navigate to the Campaigns screen. Drill down into a campaign and check the **Programs** or **Program Stages** views to see the sequence of steps and channel actions defined for the campaign.

Lead generation and nurturing

The Lead Management process focuses on converting raw inquiries into qualified sales opportunities by efficiently routing and tracking prospects.

The Lead Management Process:

  • Capture and Qualification: Leads are captured from various sources. **Qualification Rules** (often declarative expressions or Workflows) are applied to determine if the lead is sales-ready.
  • Routing (Assignment Manager): Qualified leads are automatically routed to the correct Sales Rep or Sales Team using **Assignment Manager**. The routing rules consider factors like territory, product interest, and Rep workload.
  • Nurturing: Leads that are **not** sales-ready are routed to the marketing department, where they are enrolled in **Nurturing Campaigns** until they become qualified.
  • Conversion: A qualified Lead is converted into an **Opportunity** and **Contact** record, transferring all necessary data to the sales process.

Lead Generation Flow

Validation: Navigate to the **Leads** screen. Manually update a Lead's status to trigger an automated action like assignment or conversion (if a workflow is configured to monitor that status change).

Managing customer support and service operations

Siebel Service Management extends the Call Center functionality to include full **Field Service** capabilities, critical for organizations that service physical assets at customer locations.

Field Service Lifecycle:

  • Asset Tracking: Service is managed against customer **Assets** (installed products). The Asset record tracks the product's serial number, configuration, and service history.
  • Dispatch and Scheduling: Service Requests are converted into **Activities** (work orders). The system automates the **dispatch and scheduling** of mobile field technicians based on technician **Skills**, **Location**, and current **Workload**.
  • Inventory: Tracks parts required for repairs, managing inventory levels in regional warehouses and on individual service vehicles.

Service Operations Diagram

Validation: Navigate to the **Service** screen, then to the **Assets** view. Here, you can see the installed products that a Service Request can be linked to.

Configuring service level agreements (SLAs)

Service Level Agreements (SLAs) define contracted performance metrics (like response or resolution time) for specific customers or service types. Siebel monitors these commitments in real-time.

SLA Management:

  • SLA Definition: SLAs are defined by linking **Service Contracts** (what the customer bought) to **Entitlements** (what they are owed). The SLA specifies time-to-respond and time-to-resolve targets.
  • Automated Monitoring: Siebel automatically starts timing an SR as soon as it's logged.
  • Breach Prevention: **Workflow Policies** are typically configured to monitor the remaining time on an SLA. If the time limit is approaching, the system triggers an **Automated Escalation** (e.g., sending an alert email to the manager).

SLA Monitoring Example

Workflow Policy Application: An SLA policy can check if `Service Request Priority = High` **AND** the condition has been True for a **Duration = 2 hours**. If so, it executes an action like `Update priority to Very High`.

Integrating Siebel CRM with ERP and other applications

Siebel **EAI (Enterprise Application Integration)** provides a comprehensive platform for connecting Siebel to external systems (ERP, legacy databases, cloud services).

Key Integration Technologies:

  • Siebel Business Services: These are reusable, callable functions that perform specific tasks. Services like **EAI Siebel Adapter** (to read/write Siebel BC data) and **EAI HTTP Transport** (to make web service calls) are used to interact with outside systems.
  • Integration Objects: A metadata definition in Siebel Tools that represents a Siebel Business Object in a flat, hierarchical structure (XML) for easy data exchange. The Integration Object defines the common language for integration.
  • Web Services: Siebel can consume and expose **SOAP** and **REST** Web Services, allowing real-time, synchronous integration with other applications.

Ensuring seamless data flow between systems

For large-scale, asynchronous integrations, using **EIM** remains the standard. For real-time flows, careful modeling and **error handling** are paramount.

Real-time Integration Pipeline:

  • Data Mapping/Transformation: Data traveling between systems is transformed using declarative mapping or **Integration Workflow Processes**.
  • Business Service Invocation: A **Workflow Process** orchestrates the entire flow, calling the **EAI Siebel Adapter** to get data and then calling a **Transport Business Service** to send the data out.
  • Error Handling: The integration workflow must include robust error steps to log and notify administrators if the external system fails or returns an invalid payload.

EAI Architecture Example

Example Flow: Event (Record Save) $\to$ Triggers Workflow $\to$ Calls **EAI Siebel Adapter** (to get data) $\to$ Calls **Transport Service** (to send data) $\to$ External System.

Using Siebel CRM on mobile devices

Siebel provides specialized client types to ensure users can access the CRM when disconnected from the network or using mobile devices.

Siebel Client Types:

  • Siebel Web Client (Web Browser): The primary client. Accesses the Siebel Enterprise via a web server (synchronous).
  • Siebel Mobile Web Client: A local executable installed on a laptop. It accesses a Local Database (`sse_data.dbf`) and a Local SRF, allowing users to work **disconnected**. Data is periodically synchronized with the server.
  • Siebel Developer Web Client: Used by developers/administrators for direct access to the Siebel database/repository.

Mobile Web Client Architecture

Data Model: The Mobile Web Client contains a subset of data defined during the Database Extract server task. The local SRF and CFG files govern its behavior.

Mobile app configuration and usage

Mobile access often requires separate configuration of screens and applets to ensure an optimal touch-friendly experience on small form factors.

Mobile Configuration:

  • Streamlined Application: Administrators may define a separate **Application Object** (in Siebel Tools) designed specifically for mobile users, including only the essential screens and views.
  • Mobile Web Templates: Mobile views and applets use specialized **Web Templates** designed for touch interaction and responsive behavior.
  • Synchronization: Mobile users synchronize their data using the **Synchronization Manager** server component, pushing local changes to the server and downloading updates from other users.
